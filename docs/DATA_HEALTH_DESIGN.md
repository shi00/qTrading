# 数据健康检查方案 (已实现) - Data Health Check Implementation

> **状态**: 已上线 (v2.4.1)
> **最后更新**: 2026-02-02
> **代码位置**: `data/cache_manager.py`, `data/data_processor.py`

## 1. 核心架构

当前的数据健康检查体系由三部分组成：
1.  **底层扫描器 (`CacheManager`)**: 负责执行 SQL 聚合查询，统计覆盖率与新鲜度。
2.  **评分引擎 (`DataProcessor`)**: 汇总多维数据，运用加权规则判定系统的整体健康色 (红/黄/绿)。
3.  **可视化报告 (`SystemTab`)**: 以图表形式向用户展示诊断结果。

---

## 2. 检查维度详解

### 2.1 维度一：市场行情时效性 (Market Timeliness)
*目标：确保每日行情此时此刻是最新的。*

*   **数据源**: `daily_quotes` vs `trade_cal` (官方日历)
*   **计算逻辑**:
    1.  获取官方日历中过去 3 年的所有交易日 (`Official_Set`)。
    2.  获取本地缓存的所有交易日 (`Local_Set`)。
    3.  计算 **滞后天数 (Lag Days)** = 官方最新日期 - 本地最新日期 (按交易日计数)。
*   **评级标准**:
    *   🔴 **RED**: 滞后 > 3 个交易日。
    *   🟡 **YELLOW**: 滞后 > 0 个交易日 (例如昨天收盘后未同步)。
    *   🟢 **GREEN**: 本地最新日期 == 官方最新日期。

### 2.2 维度二：深度数据覆盖率 (Deep Coverage)
*目标：确保 5000+ 只股票都有配套的基本面数据。*

*   **基准分母**: `stock_basic` 中状态为 `L` (上市) 的股票总数 (约 5300+)。
*   **扫描范围 (9大核心表)**:

| 表名 | 中文名 | 关键时间字段 | 新鲜度阈值 (Freshness Threshold) |
| :--- | :--- | :--- | :--- |
| `financial_reports` | 基础财报 | `end_date` | 180 天 |
| `fina_forecast` | 业绩预告 | `end_date` | 180 天 |
| `fina_mainbz` | 主营业务 | `end_date` | 365 天 |
| `pledge_stat` | 股权质押 | `end_date` | 30 天 |
| `repurchase` | 股份回购 | `ann_date` | 90 天 |
| `dividend` | 分红送转 | `ann_date` | 365 天 |
| `margin_daily` | 融资融券 | `trade_date` | 5 天 |
| `suspend_d` | 停牌信息 | `trade_date` | 5 天 |
| `limit_list` | 涨跌停榜 | `trade_date` | 5 天 |

*   **指标定义**:
    *   **覆盖率 (Coverage Ratio)**: `(该表中出现的不同 ts_code 数量) / (总上市股票数)`
    *   **新鲜度 (Freshness Ratio)**: `(该表中时间 > 阈值的 ts_code 数量) / (总上市股票数)`

### 2.3 评分卡逻辑 (Scorecard Logic)

系统的最终红绿灯状态由以下规则**按优先级**决定：

1.  **🔴 RED (严重警告)**
    *   **行情严重滞后**: `Lag Days > 3`
    *   **财报严重缺失**: `financial_reports` 覆盖率 < 90%
    *   **官方日历失效**: 无法获取交易所日历

2.  **🟡 YELLOW (一般警告)**
    *   **行情轻微滞后**: `Lag Days > 0`
    *   **财报轻微缺失**: `financial_reports` 覆盖率 < 98%
    *   **关键表全空**: 任意核心表 (除停牌表外) 的覆盖率为 0% (提示可能该项数据同步失败)。

3.  **🟢 GREEN (健康)**
    *   不满足上述任何警告条件。

## 3. UI 呈现

*   **状态图标**: 大号红/黄/绿盾牌图标。
*   **行情看板**: 显示具体滞后天数（红色高亮异常）。
*   **明细列表**:
    *   **进度条**: 每一行的长度代表该数据的覆盖率。
    *   **颜色编码**: 绿色 (>98%) / 橙色 (>90%) / 红色 (<90%)。
    *   **时钟标记**: 蓝色时钟代表该数据“足够新鲜”，灰色代表大多是旧数据。

## 4. 讨论与优化点 (Discussion Points)

1.  **新鲜度阈值的动态性**: 目前 `financial_reports` 固定为 180 天。在财报披露空窗期（如8月），此阈值可能过于宽松；而在披露密集期（4月），此阈值可能需要收紧。
2.  **冷门数据表处理**: 像 `repurchase` (回购) 并非所有公司都有，其覆盖率本来就不可能是 100%。目前代码一视同仁计算覆盖率，可能导致 UI 上看起来“很低”。是否需要引入“预期覆盖率”？
3.  **自动修复**: 检测到 `Lag Days > 0` 时，是否应该在该界面直接提供 "Fix Only Quotes" 按钮？

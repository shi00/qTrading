# AI 问题修复专项 Profile

> **加载方式**：由 [core-protocol.md §3 风险信号触发表](./core-protocol.md#3-风险信号--专项-profile-触发表) 按需读取。每个专项只补充核心协议的增量要点，不重复通用协议。
>
> **与项目宪法的关系**：项目特定规则（CLAUDE.md §3 红线 / §4 架构边界）优先于本文件通用建议。

## 1. 测试失败

- **先判断性质**：产品缺陷、测试缺陷、环境问题还是不稳定测试（flaky）。
- 单独运行失败测试，再检查与测试顺序、时间、随机数和共享状态的关系。
- 不以重跑成功掩盖 flaky test；修复不稳定性时记录失败率，并在足够重复次数下验证。
- 只有预期行为确实改变时才更新断言，并同步权威契约（需求/API/schema/调用方依赖）。
- 区分测试失败与被测代码失败：mock 过度会掩盖真实事务、序列化或网络语义。

## 2. 构建、依赖和环境

- 对比锁文件、运行时、工具链、系统架构和环境变量；使用干净环境验证，避免本地缓存造成假成功。
- 不随意删除锁文件或升级全部依赖；区分代码修复与仅对某台机器有效的环境绕过。
- 新增或升级依赖前检查：必要性、来源、许可、已知漏洞和破坏性变化（参考 [docs/guides/dependency-management.md](../guides/dependency-management.md)）。
- 构建失败先定位阶段（编译/链接/打包/资源收集），再判断是代码问题、配置问题还是环境问题。

## 3. 性能

- **先定义基线**：指标、负载、数据规模、硬件和可接受阈值。
- 用 profile、trace、查询计划或资源指标定位瓶颈，不靠猜测优化。
- 每次只验证一个主要优化假设；同时检查结果正确性、尾延迟、内存和退化场景。
- 没有可重复基准，不宣称性能已提升；优化不能引入精度损失或资源泄漏。
- 项目特定：UI 主循环阻塞（R16）必须通过 `ThreadPoolManager` 提交；慢操作告警见 `utils/log_decorators.py`。

## 4. 并发和间歇

- 记录线程、任务、事件顺序、时钟和随机种子；检查共享状态、原子性、锁顺序、可见性和取消传播。
- 使用高并发、重复运行、竞态检测和调度扰动提高复现率。
- 不用延时或扩大超时作为唯一修复；验证重复、乱序、超时、重启和响应丢失场景。
- **项目特定红线**：
  - R2 禁止吞没 `asyncio.CancelledError`（必须 `raise`）
  - R11 禁止跨循环复用 `asyncio.Event/Lock`（必须通过 `get_loop_local()` 获取）
  - R16 禁止在 Flet 事件处理器中同步执行 IO/CPU 密集任务
- 涉及异步/并发问题时，必须考虑事件循环归属、线程归属、取消传播三个维度（CLAUDE.md §1.7）。

## 5. 数据与迁移

- 在操作前备份或建立可恢复点；不直接在生产数据上试错。
- 明确权威数据源、约束、事务和中间状态；回填应可暂停、恢复、限速和幂等。
- 验证新旧应用与新旧 schema 共存；结果未知时先核对状态，禁止盲目重试写入。
- **项目特定**：
  - schema 变更必须生成 Alembic 迁移，验证 `upgrade head` + `alembic check`（CI 验证 `downgrade base` → `upgrade head`）
  - R8 批量写入必须用 `_save_upsert()`，禁止 `_write_db(is_many=True)`
  - R12 新增表必须更新 `data/data_dictionary.py` 的 `TABLE_DEFINITIONS`
  - R17 禁止 SQL 保留字作字段名，必须用 ORM `name=` 映射

## 6. 安全

- 限制漏洞细节和敏感证据的传播范围；按攻击入口、信任边界、数据流和权限路径定位根因。
- 修复后验证原攻击路径、变体和正常业务路径；检查同类 CWE 或相同反模式。
- 不在日志、测试、提交或报告中暴露密钥和真实敏感数据。
- 需要时遵循漏洞披露、补丁和密钥轮换流程。
- **项目特定红线**：
  - R4 asyncpg 原生查询必须用 `$1, $2, ...` 占位符，禁止 `%s`
  - R9 日志/异常必须经 `DataSanitizer` 脱敏
  - R10 禁止硬编码 API Key / DB 密码，必须从 `keyring` 或环境变量读取

## 7. 外部服务与生产环境不可用

当必要依赖、凭据、硬件或网络不可用时：

- 不伪造"已通过"；使用契约测试、mock、录制响应或静态证据完成可验证部分。
- 清楚标注模拟与真实环境的差异；给出目标环境需要执行的最小验证步骤。
- 将最终状态标为"部分验证"或"待环境验证"。
- **项目特定**：Tushare/LiteLLM/DB 等外部 IO 方法必须挂 `@log_async_operation` 或 `@track_performance()`；外部服务降级行为应在文档中说明。

## 8. 生产事故（特殊顺序）

生产事故遵循与常规修复不同的顺序：

```text
检测与定级
  → 限制影响
  → 保存证据
  → 建立时间线
  → 诊断根因
  → 安全修复
  → 分阶段验证
  → 复盘与预防复发
```

- 缓解和根因分析可以并行，但不得为追求完整根因而延误必要止损。
- 不得在未保存证据前随意重启、清理或覆盖现场。
- 复盘应形成无责复盘记录，包含根因、促成因素、逃逸原因和后续预防项。
- 高风险生产变更须获得明确授权，AI 不擅自执行部署、回滚或数据修复。

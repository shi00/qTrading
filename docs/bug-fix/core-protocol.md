# AI 问题修复核心协议

> **与项目宪法的关系**：[CLAUDE.md](../../CLAUDE.md) §3 红线（R1~R18）/ §4 架构边界是硬约束；本协议是修复方法论，提供系统化的执行流程。
>
> **冲突优先级**：法律法规 > 已批准需求/契约 > 架构/API/数据契约 > 仓库规则/项目质量门禁 > 本协议通用建议。
>
> **加载方式**：AI 日常只加载本文件；触发 §3 风险信号时 Read [profiles.md](./profiles.md)；需要详细方法/模板/参考实践时 Read [appendix.md](./appendix.md)。
>
> **与检视指南的关系**：本协议与 [docs/reviews/ai-review.md](../reviews/ai-review.md) 互补——检视针对变更质量，修复针对问题解决。

## 1. 核心原则

1.1 **先证据后结论** — 不因报错文本、Issue 描述或用户推测直接定根因；区分事实、假设、推断和待确认信息。
1.2 **先理解后修改** — 修改前理解预期/实际行为、所在层与执行路径、调用方、数据、状态、副作用与信任边界。
1.3 **修根因不掩症状** — 禁止吞错、删断言、跳测试、无依据重试、伪造默认值、关闭门禁制造假成功。
1.4 **最小但完整** — 根因修正 + 回归测试 + 必要 schema/文档/监控同步 + 同类缺陷模式有限检查。
1.5 **验证不宣称** — 实际运行测试/构建/静态检查并报告命令与结果；"应该可以"不等于已修复。
1.6 **保护现场与用户工作** — 不覆盖/回滚/提交用户已有改动；不提交临时文件、日志、凭据和生成物。

## 2. 六状态门执行协议

### S1 调查（Investigate）
- 明确预期行为、实际行为、影响范围、严重度、验收标准与权限边界
- 读取项目规则（CLAUDE.md §3/§4）、相关契约、调用方、测试说明
- 检查 `git status`，识别并保护用户已有改动
- 归一化为标准问题陈述（模板见 [appendix.md §B](./appendix.md#b-输入契约)）

### S2 复现（Reproduce）
- 优先固化为自动化测试、可重复脚本或最小样例
- 间歇问题记录失败率、并发度、时序、随机种子与资源状态
- 无法复现时不盲改：对比正常/异常环境、增加可移除诊断、收窄触发条件、声明当前置信度
- 禁止用反复试改代码代替诊断

### S3 定因（Diagnose）
- 假设驱动：维护假设账本（模板见 [appendix.md §A.1](./appendix.md#a1-假设账本模板)），每次只检验一个主要变量
- 区分症状、直接原因、关键因果因素、促成因素、逃逸原因（五要素见 [appendix.md §A.2](./appendix.md#a2-因果分析五要素)）
- 因果结论须能解释全部关键症状、触发条件与逃逸原因，且有代码/运行时/实验依据
- 不强求唯一根因，不以"人为失误"作为分析终点

### S4 修复（Fix）
- **先建立失败证据**：新增或确认能稳定失败的测试，在未修复代码上运行确认失败
- 实施最小完整修复：只改根因与必要验证相关文件，遵循现有架构、类型、错误处理与命名约定
- 禁止为让测试通过而削弱断言、标记跳过、过度 mock 或改测试迎合实现
- 发现无关缺陷记录为范围外风险，不擅自扩大修复

### S5 验证（Verify）
- 分层验证（从快到慢）：目标复现 → 回归测试 → 邻近测试 → 静态检查 → 构建验证 → 集成/E2E → 全量回归 → 目标环境
- 验证范围由影响面决定，不能只跑新增测试，也不无理由跑全量
- **反证**：原复现是否偶然通过？测试是否真经修复路径？是否只解一个样例？是否破坏上下游契约？是否引入新的安全/注入/数据泄露风险？
- 无法执行的检查须写明：未执行项、原因、替代证据、待人工确认风险

### S6 交付（Deliver）
- 已完成与未验证分开报告，不混淆
- 至少说明：根因+证据、修复内容+机制、变更文件+影响范围、已运行验证+结果、未验证项+残余风险+人工待办
- 高影响或重复问题：搜索同类缺陷模式、更新测试/规则/监控、记录根因+促成+逃逸、固化到 CI 或自动化门禁

## 4. 状态门判定

### 4.1 可完成交付（同时满足）
问题与验收标准明确；根因有足够证据；最小完整修复已实施；风险相称的回归已通过；未引入安全/数据/兼容性退化；已披露限制与残余风险。

### 4.2 应停止并请求信息/决策
权威需求或契约冲突；缺少会实质改变方案的产品/架构决策；无法区分目标改动与用户已有改动；多方案存在不可替代的用户决策权衡；必要步骤需未获授权的生产/数据写入。

### 4.3 应升级人工专家
正在利用的安全漏洞/监管/重大隐私事件；可能不可逆数据损坏；需生产紧急变更或大规模回滚；根因跨多团队/供应商且证据冲突；AI 无法安全获得必要环境/权限/专业判断。

### 4.4 允许部分交付（须准确标注状态）
仅诊断完成 / 临时缓解完成永久修复待办 / 代码修复完成目标环境验证待办 / 无法判断需补充指定证据。

## 5. 反模式快速自查

- 吞错、删断言、跳测试、关闭门禁制造假成功
- 试错式补丁（连续改多处无法解释原因）
- 只修错误出现位置（异常抛出点 ≠ 根因）
- 症状缓解冒充根因修复（重启、清缓存、扩大重试）
- 修改测试迎合实现
- 过度修复（借机重构、升级依赖、改公共接口）
- 只跑局部测试，不检查调用方与契约
- 无证据宣称完成（"应该可以""逻辑上没问题"）
- 泄露现场（完整日志、数据、截图、`.env` 入库，或清理了事故证据）
